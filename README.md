# VPN Checker

Автоматизированный агрегатор, тестер и дистрибьютор прокси-серверов с **распределённой архитектурой** (менеджер + воркеры).  
Проект поддерживает множество протоколов (VLESS, VMess, Trojan, Shadowsocks, Hysteria2) благодаря использованию ядра [sing-box](https://sing-box.sagernet.org/).

> **Статус**: Активная разработка. Недавно исправлены критические баги парсинга, добавлена система банов и standalone-режим для воркеров.

## Архитектура

Система разделена на два компонента:

### 🖥️ Менеджер (Master Panel)
Веб-панель управления, которая **не выполняет тестирование прокси**. Её задачи:
- Скачивание и парсинг подписок с прокси
- Хранение и дедупликация прокси-ссылок
- Раздача прокси-ссылок воркерам через API
- Агрегация результатов тестирования от воркеров
- Раздача лучших прокси через webhook (для VPN-клиентов)
- Управление настройками, подписками и тестовыми URL

### ⚙️ Воркер (Worker Node)
Выполняет всю работу по тестированию прокси. Поддерживает два режима запуска:

**Docker-режим:**
- Headless Docker-контейнер с автоматической установкой sing-box
- Быстрая развёртывание

**Standalone-режим:**
- Запуск без Docker (LXC-контейнеры, виртуальные машины, bare metal)
- Требует Python 3.11+ и установки sing-box вручную (скрипт `setup.sh`)

Функции воркера:
- Получает список прокси от менеджера
- Тестирует каждый прокси (TCP ping + тестовые URL через sing-box)
- Опционально измеряет скорость (download/upload)
- Отправляет результаты обратно менеджеру
- **НЕ теряет "хорошие" прокси** — даже если они исчезают из первоисточников
- **Автоматический бан** — не прошедшие тесты прокси банятся на настраиваемое время

## Особенности
- **Распределённая архитектура**: Менеджер не тестирует — все тесты выполняются воркерами из разных регионов.
- **Мультипротокольность**: VLESS, VMess, Trojan, Shadowsocks, Hysteria2.
- **Глобальный консенсус**: Прокси ранжируются по числу воркеров, которые их подтвердили.
- **Авто-обновление**: Планировщик периодически скачивает подписки, воркеры автоматически подхватывают новые прокси.
- **Стойкость к DPI**: Встроены проверки для обхода ТСПУ и геоблокировок (Instagram, YouTube, ChatGPT, RuTracker и др.).
- **Zero-Downtime Webhook**: Простая раздача лучших прокси в VPN-приложения.
- **Web-интерфейс**: FastAPI + Jinja2 + Tailwind CSS с Dark Mode / Glassmorphism UI.
- **Сохранение хороших прокси**: Прокси, прошедшие тесты, никогда не теряются — даже если исчезают из первоисточников.

- **Система банов по консенсусу**: Прокси банится только после **N последовательных неудачных проверок** (по умолчанию 3) и только если он **провалился на всех подключённых воркерах одновременно**.
- **Управление подписками**: Можно отключать опрос источников без удаления, видеть количество конфигов в каждой подписке.

## Требуемые ресурсы (Май 2026)

Система оптимизирована для работы с большими списками прокси (75k+).

### Менеджер (VPS)
- **ОС**: Любой Linux (Ubuntu, Debian, Rocky и др.)
- **Docker и Docker Compose**
- Минимально: 1 vCPU, 512MB RAM (рекомендуется 1GB для больших баз), 5GB SSD
- **Не требуется** sing-box — менеджер не тестирует прокси

### Воркер (VPS)
- **ОС**: Любой Linux
- **Два режима:**
  - **Docker**: Docker и Docker Compose (рекомендуется)
  - **Standalone**: Python 3.11+, sing-box, без Docker
- Минимально: 1 vCPU, 512MB RAM, 5GB SSD

## Развертывание менеджера

1. **Клонируйте репозиторий**:
```bash
git clone https://github.com/GoGoButters/VLESS-Checker.git
cd VLESS-Checker
```

2. **Сборка и запуск контейнера**:
```bash
docker compose up -d --build
```

3. **Доступ к панели управления**:
```
http://<ВАШ_IP_АДРЕС>:8000
```
- **Пароль по умолчанию**: `admin`
- *(Обязательно смените в разделе **Settings** → Security)*

## Развертывание воркера

Выберите способ развертывания:

---

### Docker-режим (рекомендуется)

На отдельном сервере:

1. **Клонируйте репозиторий**:
```bash
git clone https://github.com/GoGoButters/VLESS-Checker.git
cd VLESS-Checker/node
```

2. **Настройте `docker-compose.yml`**:
```yaml
environment:
  - MASTER_URL=http://<IP_МЕНЕДЖЕРА>:8000
  - NODE_TOKEN=<токен из Settings менеджера>
  - NODE_NAME=my-worker-1
  - NODE_REGION=Russia
```

3. **Запустите воркер**:
```bash
docker compose up -d --build
```

---

### Standalone-режим (без Docker)

Подходит для LXC-контейнеров, виртуальных машин или bare metal.

1. **Клонируйте репозиторий**:
```bash
git clone https://github.com/GoGoButters/VLESS-Checker.git
cd VLESS-Checker/node
```

2. **Запустите установку** (установит Python, sing-box, зависимости):
```bash
chmod +x setup.sh
./setup.sh
```
Скрипт автоматически:
- Установит sing-box v1.11.4
- Настроит Python virtual environment
- Установит зависимости (httpx, pydantic-settings)

3. **Скопируйте и отредактируйте конфигурацию**:
```bash
cp .env.example .env
nano .env
```
Отредактируйте `.env`:
- `MASTER_URL` — адрес менеджера (http://IP:8000)
- `NODE_TOKEN` — токен из **Settings → Node Management** в панели
- `NODE_NAME` — имя воркера
- `NODE_REGION` — регион
- `POLL_INTERVAL_S` — интервал опроса (по умолчанию 60с)

4. **Запустите воркер**:
```bash
./run.sh
```

Воркер автоматически зарегистрируется на менеджере и начнёт тестирование прокси.

**Отладка standalone-режима:**
- Логи выводятся в консоль
- Для остановки: `Ctrl+C`
- Для перезапуска: `./run.sh`

## Настройка системы

В панели управления (**Settings**):
- **Worker Test Parameters** — Ping-порог, concurrency, таймауты (эти настройки передаются воркерам)
- **Auto-Fetch Scheduler** — Интервал автоматического скачивания подписок (минuty, 0 = отключено)
- **Webhook** — Секретный путь и лимит прокси для раздачи
- **Node Management** — API-токен для воркеров, лимит прокси на воркер

- **Ban Management**:
    - **Ban Duration (hours)** — Срок бана (0 = отключено)
    - **Ban After N Failures** — Сколько раз подряд прокси должен провалиться на **всех** нодах перед баном
- **Global Consensus Sub** — Лимит для глобальной подписки (ранжирование по кросс-нодовому консенсусу)
- **Subscriptions** — Управление источниками: включение/отключение без удаления, отображение количества конфигов
- **Security** — Смена пароля администратора


## Система банов по кросс-нодовому консенсусу

В отличие от простых чекеров, эта система использует **интеллектуальный бан**, чтобы избежать ложных срабатываний из-за проблем на одной конкретной ноде.

### Как это работает:
1. **Проверка на всех нодах**: Воркеры отправляют результаты (успех/провал) менеджеру.
2. **Консенсус**: 
    - Если прокси **прошёл хотя бы на одной ноде** → он считается рабочим, счётчик провалов сбрасывается в 0.
    - Если прокси **провалился на всех подключённых нодах** одновременно → счётчик `consecutive_failures` увеличивается на 1.
3. **Порог бана**: Только когда счётчик достигает значения **N** (настраивается в Settings, по умолчанию 3), прокси получает метку `banned_until`.
4. **Фильтрация**: Забаненные прокси не отправляются воркерам и не попадают в webhook до истечения срока бана.

### Преимущества:
- Если у одного воркера проблемы с сетью — он не забанит всю базу.
- Бан срабатывает только на стабильно нерабочие конфиги.
- Вы сами решаете "степень доверия" через настройку N-провалов.

## Исправление страницы Proxies

Ранее страница `/proxies` имела проблемы с отображением — вся информация сжималась в первую колонку.

### Что исправлено:
- **Удалена дублирующая таблица** (конфликт слияния)
- **Каждая колонка в своём месте**: Config, Nodes, Tests, Ping, DL, UL, Score, Tested
- **Config колонка** теперь показывает только ссылку на прокси (обрезается до 80 символов)
- **Остальные данные** (Nodes, Tests, Ping, скорость, рейтинг) отображаются в отдельных колонках

### Колонки таблицы:
| Колонка | Описание |
|---|---|
| Config | Ссылка на прокси (обрезанная до 80 символов) |
| Nodes | Количество воркеров, подтвердивших прокси |
| Tests | Пройдено/Всего тестов |
| Ping | Лучший пинг (цвет: зелёный <200мс, жёлтый <500мс, красный ≥500мс) |
| DL | Средняя скорость скачивания (KB/s) |
| UL | Средняя скорость отдачи (KB/s) |
| Score | Рейтинг (на основе пинга, тестов и скорости) |
| Tested | Дата последней проверки |

**Важно для больших списков прокси (75k+):**
При проверке огромного количества конфигов (75 000+) может возникнуть ошибка `errno 24: Too many open files`.
**Решение:**
1. Увеличьте лимит файловых дескрипторов на ноде: `ulimit -n 65536`.
2. Для Docker-нод добавьте в `docker-compose.yml`:
   ```yaml
   ulimits:
     nofile:
       soft: 65536
       hard: 65536
   ```
3. Убедитесь, что в `node/config.py` настройка `concurrent_checks_limit` (по умолчанию 50) не завышена.

База данных сохраняется в volume `vpn_data` и не стирается при обновлении.
